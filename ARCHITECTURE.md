# ServerlessAgent — Architecture & Improvement Roadmap

## What's been built

An event-driven AI personal assistant running on AWS Lambda. The compute is fully stateless — every Lambda invocation is independent. All state lives in DynamoDB.

```
Client
  │
  ▼
API Gateway (POST /webhook, GET /status/{jobId})
  │
  ▼
Middleware Lambda             ← validates API key, routes text vs voice, manages transcription jobs
  │                │
  ▼                ▼ (async)
Agent Lambda    Transcriber Lambda
  │                │
  ▼                ▼
Bedrock (Nova)  AWS Transcribe + S3
  │
  ▼
DynamoDB (session memory, notes, expenses, transcription jobs)
```

---

## The core challenge: stateless compute + stateful agent

Serverless means Lambda has zero memory between invocations. But a useful AI assistant must remember things. The current solution: everything lives in DynamoDB, loaded fresh on every invocation.

**This is the right approach.** But *what* you store, *how much* you store, and *how you structure it* determines how smart your agent feels.

---

## Current memory model (and its limits)

Right now there is one type of state: `session.messages[]` — a raw list of every message in the conversation, trimmed to the last 20 turns.

```
DynamoDB item: session#<sessionId>
  messages: [
    { role: 'user',      content: 'Hi, I am Max, I live in Berlin' },
    { role: 'assistant', content: 'Nice to meet you Max!' },
    { role: 'user',      content: 'Schedule a meeting tomorrow' },
    ...
  ]
```

**Problems with this model:**

1. **Amnesia at turn 21** — when you hit the trim limit, the oldest messages are deleted blindly. "I live in Berlin" disappears exactly the same as irrelevant small talk.
2. **No long-term facts** — if the user told the agent their timezone, boss's name, or email address in turn 1, that is gone by turn 25. The agent becomes a stranger again.
3. **Growing Bedrock cost** — the full history is sent to the model on every turn. Cost grows with conversation length, even for simple requests.
4. **Static system prompt** — the agent is initialized identically for every user on every turn. It knows nothing about who it is talking to until they tell it again.

---

## Recommended: Three-Layer Memory Architecture

The fix is to separate state into three layers with different lifetimes and different roles.

```
┌─────────────────────────────────────────────────────────────┐
│  LAYER 1: User Profile  (permanent, never trimmed)          │
│  What the agent knows about this user permanently.          │
│  DynamoDB key: profile#<userId>                             │
│                                                             │
│  {                                                          │
│    name: "Max",                                             │
│    timezone: "Europe/Berlin",                               │
│    language: "German",                                      │
│    calendarId: "max@gmail.com",                             │
│    contacts: { "boss": "anna@company.com" },                │
│    preferences: { meetingDuration: 60 }                     │
│  }                                                          │
│                                                             │
│  → Injected into the system prompt on every turn.           │
│  → Updated by the agent when it learns something new.       │
├─────────────────────────────────────────────────────────────┤
│  LAYER 2: Episodic Summary  (weeks, regenerated)            │
│  A compressed summary of older conversation history.        │
│  DynamoDB key: summary#<userId>                             │
│                                                             │
│  "Past week: Scheduled 3 meetings. Researched AI laws.      │
│   Total expenses: €240 (Food €80, Transport €60)."          │
│                                                             │
│  → Generated by a background Lambda on a cron schedule.     │
│  → Included as context at the start of each conversation.   │
│  → Replaces dozens of raw message turns with a few lines.   │
├─────────────────────────────────────────────────────────────┤
│  LAYER 3: Working Memory  (current session, ~5 turns)       │
│  The last few raw message pairs from this conversation.     │
│  DynamoDB key: session#<sessionId>                          │
│                                                             │
│  → Keeps Bedrock context small and cheap.                   │
│  → TTL: 7 days — old sessions expire automatically.         │
└─────────────────────────────────────────────────────────────┘
```

With this model, every Bedrock call sends:
- A **rich system prompt** that knows who the user is (Layer 1)
- A **brief summary** of recent activity (Layer 2)
- Only **the last 5 raw turns** (Layer 3, not 20)

The agent feels like it truly knows the user, costs less per call, and important facts survive indefinitely.

## How to implement Layer 1: User Profile

### Step 1 — DynamoDB schema

```typescript
// profile#<userId> item in session table (or separate ProfileTable)
interface UserProfile {
  sessionId: string;           // "profile#<userId>"
  name?: string;
  timezone?: string;           // e.g., "Europe/Berlin"
  language?: string;           // e.g., "German"
  calendarId?: string;
  contacts?: Record<string, string>;
  preferences?: {
    meetingDuration?: number;  // minutes
    currency?: string;
  };
  updatedAt: number;
}
```

### Step 2 — Dynamic system prompt

Instead of a static string, build the system prompt at runtime:

```typescript
async function buildSystemPrompt(userId: string): Promise<string> {
  const profile = await loadUserProfile(userId);
  const profileSection = profile
    ? `USER PROFILE:\n${JSON.stringify(profile, null, 2)}`
    : 'No profile yet — learn about the user and update it as you go.';

  return `${BASE_SYSTEM_PROMPT}\n\n${profileSection}`;
}
```

### Step 3 — update_profile skill

Add a skill that the agent calls when it learns something new about the user:

```typescript
export const updateProfileSkill: Skill = {
  definition: {
    name: 'update_user_profile',
    description: "Update persistent information about the user (name, timezone, preferences, contacts). Call this whenever you learn something new that should be remembered permanently.",
    inputSchema: { ... },
  },
  execute: (input) => updateProfile(input),
};
```

Add to system prompt: *"When you learn a fact about the user that should be remembered forever (name, timezone, preferences, contacts), call update_user_profile immediately."*

### Step 4 — reduce working memory to 5 turns

With a rich profile and summary, 5 turns of raw history is enough. Change `MAX_TURNS = 5` in `dynamo-memory.ts`. This cuts Bedrock token costs by ~75% on long conversations.

## How to implement Layer 2: Episodic Summary

### Option A — EventBridge cron (recommended)

Create a fourth Lambda `summarizer` that runs nightly:

1. Reads all session items updated in the past 24 hours
2. For each user, sends the raw messages to Bedrock: *"Summarize this conversation history in 3–5 sentences."*
3. Writes the summary to `summary#<userId>` in DynamoDB
4. The session messages older than 5 turns are then safe to trim aggressively

CDK setup:
```typescript
const rule = new events.Rule(this, 'DailySummary', {
  schedule: events.Schedule.cron({ hour: '2', minute: '0' }), // 2am daily
});
rule.addTarget(new targets.LambdaFunction(summarizerLambda));
```

### Option B — inline summarization (simpler, no new Lambda)

When `trimMessages()` is about to discard old turns, summarize them first:

```typescript
// In dynamo-memory.ts save():
if (messages.length > MAX_MESSAGES) {
  const toSummarize = messages.slice(0, messages.length - MAX_MESSAGES);
  const newSummary = await summarizeWithBedrock(toSummarize, existingSummary);
  // Store summary separately, keep only recent messages
}
```

## How to implement session TTL

Sessions that were never resumed should expire automatically. Add this to `dynamo-memory.ts`:

```typescript
// In save():
Item: {
  sessionId,
  agentState: { ... },
  ttl: Math.floor(Date.now() / 1000) + (30 * 24 * 60 * 60), // 30 days
}
```

Make sure `timeToLiveAttribute: 'ttl'` is set on the table in `db-stack.ts` (already done for the session table).

## Remaining architectural improvements

### Multiple session threads

Right now every user has one session. Consider allowing named threads:

```
POST /webhook  { sessionId: "work", message: "..." }
POST /webhook  { sessionId: "personal", message: "..." }
```

The middleware already echoes `sessionId` back — the client just needs to persist it.

### Middleware timeout alignment

API Gateway HTTP API has a **29-second hard timeout**. The middleware and agent Lambda are both set to 60 seconds. For text requests this is fine (agent runs inside Lambda, not behind API GW directly). But ensure the agent's average response time stays under 25 seconds for comfortable use.

### Secrets Manager caching

The `loadSecrets()` function currently uses a module-level `loaded` boolean, which works correctly within a single warm Lambda container. This is the right approach — do not refetch on every turn.

## Adding a new skill — checklist

```
1. Create services/agent/src/skills/my-skill.ts
   - Implement the function
   - Export a Skill object: { definition, execute }

2. Add the import + entry to services/agent/src/skills/index.ts

3. Done. agent.ts auto-discovers it from the registry.
```

No other files need to change.

## DynamoDB table summary

| Table | Key pattern | Contents | TTL |
|---|---|---|---|
| `SessionTable` | `session#<id>` | Conversation messages | 30 days (add to save) |
| `SessionTable` | `transcription#<jobId>` | Async audio job state | 1 hour (already set) |
| `NotesTable` | `note#<key>` | User notes | None |
| `ExpensesTable` | `expense#<timestamp>` | Expense records | None |
| *(future)* SessionTable | `profile#<userId>` | User profile | Never |
| *(future)* SessionTable | `summary#<userId>` | Episodic summary | None |
